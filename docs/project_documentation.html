1. Project Overview

The AI-Driven Adaptive Diagnostic Engine is a professional-grade adaptive testing system that determines a student's proficiency level by dynamically selecting questions based on their previous answers. Unlike traditional fixed-difficulty tests, this system adapts in real-time — delivering harder questions when the student answers correctly and easier ones when they struggle.

Core Idea: The system uses a mathematically rigorous algorithm (Item Response Theory) to estimate student ability with maximum precision in minimal questions — typically just 10 questions to get an accurate proficiency assessment.

Key Features

  • Adaptive Question Selection — Questions chosen to maximize measurement precision (Fisher Information)
  • IRT-Based Ability Estimation — Rasch Model with Maximum Likelihood Estimation updates
  • 22 GRE-Style Questions — Across 5 topics: Algebra, Vocabulary, Geometry, Reading Comprehension, Quantitative Reasoning
  • AI-Powered Study Plans — Groq LLM (LLaMA 3.3 70B) generates personalized 3-step study plans
  • Session Tracking — Full history of responses, ability progression, and analytics
  • Interactive API Docs — Swagger UI auto-generated at /docs

6. Adaptive Algorithm — Item Response Theory

Why IRT? Item Response Theory is the gold standard for adaptive testing, used by the GRE, GMAT, and other major standardized tests. It provides a mathematically rigorous way to estimate ability and select optimal questions.

6.1 The Rasch Model (1-Parameter IRT)

The probability that a student with ability θ answers a question of difficulty b correctly:

Item Characteristic Curve (ICC)
P(correct | θ, b) = 1 / (1 + e-a(θ - b))
ParameterDescriptionValue/Range
θ (theta)Student ability — latent trait0.05 – 0.95
bItem difficulty0.10 – 1.00
aDiscrimination constant (fixed)4.0

Key property: When θ = b, the probability is exactly 50%. This is the point of maximum information.

6.2 Ability Update (After Each Response)

After each answer, the student's ability estimate is updated using a gradient-ascent step on the log-likelihood:

Maximum Likelihood Estimation Update
θnew = θold + η × (y - P(θ, b))
VariableMeaning
y1 if correct, 0 if incorrect
ηLearning rate = 0.15
y - PResidual: positive = outperformed model, negative = underperformed
Example: Student at θ=0.5 answers a b=0.5 question correctly.
P(correct) = 0.50, residual = 1.0 - 0.50 = 0.50
θ_new = 0.5 + 0.15 × 0.50 = 0.575 (ability increases)

6.3 Question Selection (Maximum Fisher Information)

The next question is chosen to maximize measurement precision:

Fisher Information Function
I(θ, b) = a² × P(θ, b) × (1 - P(θ, b))

Fisher Information peaks when b = θ (question difficulty matches student ability). The engine:

  1. Fetches unanswered questions near the student's current ability
  2. Computes Fisher Information for each candidate
  3. Picks the question with the highest information value

6.4 Final Ability Computation

After all 10 questions, the final ability is recomputed from scratch using iterative Newton-Raphson MLE over ALL responses. This is more stable than the incremental per-question updates.

7. AI Insights — Groq LLM Integration

After completing 10 questions, the system sends a structured performance summary to the Groq API (LLaMA 3.3 70B Versatile model) and receives a personalized 3-step study plan.

Data Sent to LLM

  • Final Ability Score — 0 to 1 scale
  • Difficulty Level Reached — Maximum difficulty answered correctly
  • Score — X/10 correct
  • Weak Topics — Topics with incorrect answers
  • Strong Topics — Topics with correct answers

Prompt Strategy

The system uses a structured prompt that instructs the LLM to act as a GRE tutor and produce a concise, actionable 3-step plan that:

  1. Targets the student's specific weaknesses
  2. Builds on their existing strengths
  3. Includes concrete practice recommendations
Caching: The study plan is generated once and cached in the UserSession document. Subsequent requests for the same session return the cached plan without additional API calls.

9. User Flow — How to Use the Application

1
Open Swagger UI
Navigate to http://localhost:8000/docs in your browser. This is the interactive API playground.
2
Start a New Session
Click POST /diagnostic/start → "Try it out" → "Execute".
Copy the session_id from the response.
3
Get Your First Question
Click GET /diagnostic/next-question → paste your session_id → Execute.
The system selects a question at difficulty 0.5 (your baseline). Note the id and options.
4
Submit Your Answer
Click POST /diagnostic/submit-answer → provide session_id, question_id, and selected_answer (the exact option text).
The response tells you if you were correct and shows your updated ability score.
5
Repeat 10 Times
Alternate between "Get Next Question" and "Submit Answer" for 10 questions.
✅ Correct → Next question is harder
❌ Incorrect → Next question is easier
Watch your ability score change after each answer.
6
Get Your Results & Study Plan
After question 10, call GET /diagnostic/session-summary.
You'll receive: final score, ability estimate, strong/weak topics, and a personalized 3-step AI study plan from Groq LLM.
Pro Tip: You can also use GET /questions/ to browse all available questions filtered by topic or difficulty, and GET /questions/topics to see all subject areas.

11. AI Log

How AI Tools Were Used

ToolUsage
GitHub CopilotPrimary development — project scaffolding, IRT algorithm implementation, Groq integration, seed data generation, API documentation

AI Accelerated Areas

  • FastAPI project structure with clean module separation
  • 22 realistic GRE-style questions with calibrated difficulties
  • IRT Rasch model implementation (ICC, Fisher Information, MLE updates)
  • Groq LLM prompt engineering for study plan generation
  • Pydantic v2 model design with proper validation
  • MongoDB async driver setup with Motor

Challenges AI Couldn't Fully Solve

  1. IRT Parameter Tuning — The discrimination constant (a = 4.0) and learning rate (η = 0.15) required manual tuning and verification through testing. AI gave reasonable starting values, but the final calibration needed human judgment to ensure ability estimates stayed within [0, 1] and remained responsive but stable.
  2. Fisher Information vs Simple Matching — Initial AI suggestion was nearest-difficulty matching. The upgrade to Fisher Information maximization (the theoretically optimal approach for CAT selection) was a deliberate architectural decision requiring IRT domain knowledge.
  3. MongoDB Index Strategy — Choosing compound indexes on (difficulty, topic) required understanding the actual query patterns of the adaptive algorithm's progressive search windows.
